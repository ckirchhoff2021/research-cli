# Skill Optimizer

用于对现有 Skill 做执行级分析和优化，不直接凭空评价 Skill，而是先运行目标 Skill、记录轨迹，再基于证据提出改进建议。

## 何时使用

当出现以下场景时使用本技能：

- 用户要求优化、改进、迭代某个 Skill
- 用户想看某个 Skill 的执行过程、工具调用链路或 trace
- 用户认为某个 Skill 效果不好，希望定位问题
- 用户需要对 Skill 做回归测试、样例评测、对比分析

## 必要输入

开始执行前，至少确认以下信息：

| 参数 | 是否必填 | 说明 |
|---|---|---|
| `skill` | 是 | 目标 Skill 的路径。若底层运行器要求传入 skills source path，则传目标 Skill 的上一级目录。 |
| `task` | 是 | 用于触发目标 Skill 的测试任务。既可以是单条自然语言任务，也可以是一个 JSON 文件路径。 |
| `trace_file` | 否 | 执行轨迹输出文件路径。若不传，脚本会自动生成 `trace_YYYYMMDD_HHMMSS.json`。 |
| `expected_output` | 否 | 期望输出或验收标准。若用户提供，可用于结果判定。 |

如果用户没有提供可执行的测试任务，先帮助用户补齐 1 到 3 条有代表性的真实样例，再开始执行。

如果 `task` 是 JSON 文件路径，当前 `harness.py` 会按数组读取，并从每个对象的 `query` 字段中提取任务后循环执行。

## 执行目标

每次执行本技能时，必须完成以下三件事：

1. 运行目标 Skill，获得真实输出
2. 记录执行轨迹，包括工具调用、关键中间步骤和最终结果
3. 基于轨迹分析 Skill 的问题，并输出结构化优化建议

## 执行流程

### 1. 准备阶段

1. 确认目标 Skill 路径、测试任务和验收标准
2. 先读取目标 Skill 的 `SKILL.md`
3. 如目标 Skill 依赖 `references/`、`scripts/` 或 `assets/`，按需补充读取

### 2. 轨迹采集阶段

使用 `scripts/harness.py` 执行测试任务：

```bash
cd skills/skill-optimizer/scripts
python harness.py \
  --skill "<target-skill-source-path>" \
  --task "<test-task-or-json-file>" \
  [--trace_file "<trace-output-file>"]
```

如果传入的是 JSON 文件，推荐格式如下：

```json
[
  {
    "query": "测试任务 1"
  },
  {
    "query": "测试任务 2"
  }
]
```

脚本会按数组顺序循环执行每个 `query`，并将所有结果汇总写入同一个 trace 文件。

执行时必须记录：

- 原始用户输入
- 多任务模式下的任务顺序
- 是否命中目标 Skill
- 工具调用名称、参数、结果摘要
- 关键中间推理或决策步骤
- 最终输出
- 错误信息和失败位置（如果失败）

### 3. 结果分析阶段

基于轨迹，从以下维度分析：

1. **触发是否正确**
   - 是否命中目标 Skill
   - 是否误触发或漏触发
   - `description` 是否足以支持正确路由

2. **流程是否合理**
   - 是否遵循目标 Skill 的预期执行顺序
   - 是否跳过了必要步骤
   - 是否存在冗余步骤

3. **工具调用是否正确**
   - 工具选择是否合理
   - 参数是否正确
   - 工具结果是否被正确使用

4. **输出是否符合预期**
   - 最终结果是否满足任务要求
   - 格式是否正确
   - 是否遗漏关键字段、路径、链接或说明

### 4. 产出优化建议

必须输出结构化优化建议，至少包含：

- 发现的问题
- 证据来源（哪一步轨迹、哪次工具调用、哪段输出）
- 建议修改的位置
- 推荐修改方式
- 预期收益

## 输出要求

至少输出两类结果：

### 1. 执行轨迹

当前 `harness.py` 默认输出为 JSON 数组，每个元素对应一个任务，结构如下：

```json
[
  {
    "task": "原始任务文本",
    "traces": [
      {
        "type": "query",
        "content": "用户原始查询"
      },
      {
        "type": "tool_call",
        "name": "工具名称",
        "args": {
          "key": "value"
        },
        "result": "工具结果摘要"
      },
      {
        "type": "thinking",
        "content": "关键中间分析"
      },
      {
        "type": "output",
        "content": "最终输出"
      }
    ]
  }
]
```

其中单条 `traces` 内部的消息结构如下：

```json
{
  "type": "tool_call",
  "name": "工具名称",
  "args": {
    "key": "value"
  },
  "result": "工具结果摘要"
}
```

### 2. 优化建议

使用结构化 Markdown 输出，例如：

```markdown
## 优化建议

### 问题 1：重复读取技能文档
- 证据：第 2 步和第 5 步重复调用 read_file 读取同一份 SKILL.md
- 影响：增加额外工具调用和等待时间
- 建议：在单轮执行中缓存已读取的技能文档内容
- 预期收益：减少重复工具调用，缩短执行耗时
```

## 结果保存

当前 `harness.py` 会将全部任务轨迹写入 `--trace_file` 指定的文件。

如果未指定 `--trace_file`，默认在当前工作目录生成： `trace_YYYYMMDD_HHMMSS.json`


## 注意事项

1. 不要在没有执行证据时直接评价 Skill 好坏
2. 不要只看最终答案，必须关注触发、流程和工具调用
3. 不要在分析阶段直接覆盖原始 Skill，先输出分析和建议
4. 如果执行失败，必须保留完整错误上下文，便于复盘
5. 如果用户未提供训练集或验证集，先做单样例执行分析，不要伪造 benchmark 结果
